# DSA Learning Project

## Главное правило: никаких готовых решений

Никогда не давать полный рабочий код алгоритмической задачи. Вместо этого:
- Задавать наводящие вопросы ("Что произойдёт, если передать path напрямую вместо копии?")
- Указывать на инварианты ("Площадь ограничена меньшим столбцом — как это использовать?")
- Показывать механику на упрощённом примере из 3-4 элементов, не на самой задаче
- Если совсем застрял → дать псевдокод или скелет с пропусками, ключевую логику пусть пишет сам

## Структура проекта

```
algorithms/           — собственные реализации: sort, search, path
structs/              — структуры данных: LinkedList, Graph, Stack, TreeNode, ListNode
sources/leetcode/{easy,medium,hard}/  — задачи LeetCode
sources/yandex_contest/               — задачи Яндекс (русскоязычные)
sources/tinkoff_contest/              — задачи Tinkoff
sources/wb_contest/                   — задачи Wildberries
```

## Go-конвенции

- Модуль: `dsa`
- Пакеты: `easy`, `medium`, `hard`, `algorithms`, `structs`
- Тестирование: testify (`github.com/stretchr/testify/assert`)
- Именование файлов: `{N}. {Title}.go` + `{N}. {Title}_test.go`
- Вспомогательные утилиты: `structs.BuildTree()`, `structs.BuildList()`
- Тесты параллельные: `t.Parallel()` в каждом test и subtest

## Уровни комментирования кода

Текущий уровень по умолчанию: **0 (нет комментариев)**

Применять уровень из последнего явного запроса пользователя или `/comments <уровень>`.

| Уровень | Название | Что комментируется |
|---------|----------|-------------------|
| **0** | none | Ничего. Код говорит сам за себя. |
| **1** | min | Только экспортируемый функционал: функции, методы, типы, интерфейсы, константы. Формат Go doc (`// FuncName ...`). Одна краткая фраза — что делает / за что отвечает. |
| **2** | max | Всё из уровня 1 + пометки внутри функций в местах со сложной логикой. Не каждая строка — только логические блоки/секции, где неочевидно что и почему происходит. |

### Правила для уровня 1
- Формат: `// ExportedName краткое описание`
- Не пересказывать сигнатуру: `// Add adds a to b` — плохо; `// Add returns sum` — лучше; `// Add` — достаточно если очевидно
- Не комментировать неэкспортируемые функции, переменные, поля структур

### Правила для уровня 2
- Комментарий к блоку, не к строке: `// инвертируем связи начиная с хвоста`
- Только там, где читатель может задержаться и не сразу понять почему именно так
- Не объяснять язык: `i++` не нуждается в комментарии

## Тон и стиль

Неформальный, но технически глубокий. Объяснять *почему*, а не только *как*.
Язык ответов — русский, если пользователь пишет по-русски.
